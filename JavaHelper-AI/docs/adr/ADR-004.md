# ADR 004: Abstração da Base de Conhecimento através de Interface

* Status: Proposto
* Data: 2026-07-19
* Autores: Time de Arquitetura
* Decisores: Time de Desenvolvimento

## Contexto e Problema

Atualmente, a classe `MockChatService` depende diretamente da implementação concreta `MockKnowledgeBase`:

```
MockChatService
        │
        ▼
MockKnowledgeBase
```

Embora esta abordagem funcione no cenário atual de testes, ela cria um acoplamento rígido que compromete a evolução futura do sistema. Questionamos integrar diferentes fontes de conhecimento:

- `OpenAIKnowledgeBase`
- `SpringAIKnowledgeBase`
- `ElasticSearchKnowledgeBase`

Com a arquitetura atual, cada nova integração exigiria modificações na classe `MockChatService`, violando o princípio do Aberto/Fechado (OCP) e gerando retrabalho significativo.

## Opções Consideradas

### Opção 1: Manter Acoplamento Direto (Status Quo)

* Prós:
  - Simplicidade imediata
  - Menor esforço inicial de desenvolvimento
  - Sem necessidade de refatoração

* Contras:
  - Alta dependência de implementação específica
  - Necessidade de modificar `MockChatService` para cada nova base de conhecimento
  - Dificuldade em testar o serviço isoladamente
  - Viola o Dependency Inversion Principle (DIP)
  - Aumenta o custo de manutenção e evolução
  - Risco de introdução de bugs ao alterar código existente

### Opção 2: Criar Interface KnowledgeBase

* Prós:
  - Desacoplamento entre serviço e implementação concreta
  - Facilita testes unitários (possibilidade de mocks)
  - Permite substituição de implementações sem alterar o serviço
  - Prepara a arquitetura para integrações futuras
  - Segue os princípios SOLID (especificamente DIP e OCP)
  - Permite injeção de dependência
  - Facilita manutenção e evolução do sistema

* Contras:
  - Esforço inicial de refatoração
  - Necessidade de adicionar uma nova abstração ao código
  - Curva de aprendizado para o time compreender a nova arquitetura
  - Possível complexidade adicional em momentos iniciais

## Decisão

Escolhemos a Opção 2 - Criar Interface KnowledgeBase porque esta decisão resolve o acoplamento atual e prepara o sistema para evoluções futuras de forma sustentável.

Esta escolha resolve o problema principal garantindo que a aplicação passe a depender da abstração (interface) e não da implementação concreta, possibilitando que novas bases de conhecimento sejam integradas sem modificar `MockChatService`.

A nova arquitetura seguirá o padrão:

```
KnowledgeBase (interface)
              ▲
              │
   ┌──────────┴──────────┐
   │                     │
   │                     │
MockKnowledgeBase   OpenAIKnowledgeBase
```

## Consequências

### Positivas (Ganhos)

* Desacoplamento do serviço da implementação específica da base de conhecimento
* Facilidade para substituir a implementação (ex: trocar Mock pelo OpenAI em produção)
* Simplificação dos testes unitários com possibilidade de criar mocks da interface
* Preparação da arquitetura para integrações futuras sem modificar código existente
* Alinhamento com os princípios SOLID (DIP)
* Melhor manutenibilidade e escalabilidade do sistema
* Redução do custo de mudança para novas integrações

### Negativas (Trade-offs / Riscos)

* Esforço necessário para refatoração do código existente
* Necessidade de garantir que todas as implementações sigam o contrato da interface
* Custo inicial de desenvolvimento para criar a abstração e adaptar o código
* Necessidade de gerenciar a injeção de dependência adequadamente
