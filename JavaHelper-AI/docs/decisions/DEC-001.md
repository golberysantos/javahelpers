DEC-001

Título

Testes fazem parte da implementação.

Descrição

No JavaHelper AI, uma funcionalidade não será considerada concluída apenas porque está funcionando manualmente.

Ela deverá possuir testes automatizados compatíveis com sua responsabilidade.

Essa decisão reforça nossa Definition of Done e a cultura de qualidade que definimos para o projeto.


Toda Sprint começará com um Sprint Planning documentado.

Isso significa que, antes de escrever código, vamos responder:

Qual é o objetivo?
Qual é o backlog?
Quais são os critérios de aceitação?
Quais artefatos esperamos produzir?
Existe alguma decisão arquitetural prevista?

Essa pequena cerimônia nos dará clareza sobre o que realmente queremos alcançar.


Quero propor um pequeno padrão para o JavaHelper AI

Gostaria que, sempre que escrevêssemos um novo teste, ele obedecesse a esta estrutura:

@Test
void deve...() {

    // Arrange

    // Act

    // Assert

}

Mesmo quando o método for pequeno.

Por quê?

Porque a consistência facilita a leitura.

Um novo membro da equipe abrirá qualquer teste e encontrará a mesma organização.

Essa é uma decisão de engenharia que vale a pena padronizar.

Essa última etapa não é obrigatória para implementar imediatamente. Ela serve para registrar ideias de melhoria enquanto o contexto ainda está fresco.


# Convenção para nossos Code Reviews.

Ao final de cada funcionalidade, responderemos estas quatro perguntas:

O comportamento está correto?
O teste realmente protege esse comportamento?
Existe algum acoplamento desnecessário?
Há alguma melhoria que podemos registrar para uma futura refatoração?

Isso cria uma cultura de evolução contínua sem cair na armadilha de querer deixar tudo perfeito na primeira versão.
